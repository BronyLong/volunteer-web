## Проектирование архитектуры веб-приложения

### 1. Общая архитектура системы

Проектирование архитектуры веб-приложения включает определение основных компонентов системы, распределение ответственности между ними и описание способов взаимодействия. Для разрабатываемого веб-приложения волонтёрской организации архитектура строится по принципу многослойной системы и ориентирована на поддержку ключевых бизнес-процессов: регистрацию и аутентификацию пользователей, управление профилями пользователей, работу с волонтёрскими мероприятиями и заявками на участие, а также управление мероприятиями со стороны координатора.

Архитектура приложения реализована по классической трёхслойной схеме и включает следующие уровни:

- **Клиентский слой (frontend)**  
  Представлен одностраничным веб-приложением (SPA), разработанным с использованием библиотеки React.js. Этот слой отвечает за отображение пользовательского интерфейса, обработку действий пользователя и отправку запросов к серверу. После первоначальной загрузки приложения дальнейшие переходы между разделами (главная страница, список мероприятий, страница мероприятия, личный кабинет, страницы управления мероприятиями координатора) выполняются без полной перезагрузки страницы, что повышает удобство и отзывчивость интерфейса.

- **Серверный слой (backend)**  
  Реализован на платформе Node.js с использованием фреймворка Express. Серверный слой обрабатывает HTTP-запросы от клиентского приложения, выполняет проверку прав доступа в соответствии с ролью пользователя (гость, волонтёр, координатор), реализует бизнес-логику системы (регистрация и аутентификация, работа с профилями, управление мероприятиями, подача и отмена заявок) и формирует ответы в формате JSON.

- **Слой хранения данных (база данных)**  
  Для постоянного хранения данных используется реляционная система управления базами данных PostgreSQL. В базе данных размещаются сведения о пользователях и их профилях, мероприятиях, заявках волонтёров на участие, а также справочная информация, необходимая для работы системы, например категории мероприятий. Серверный слой обращается к базе данных посредством SQL-запросов, обеспечивая целостность и согласованность данных.

Такое разделение на слои позволяет чётко разграничить ответственность между клиентской частью, серверной логикой и хранилищем данных, упрощает сопровождение и дальнейшее развитие системы.

### 2. Основные подсистемы и модули

В рамках выделенных слоёв можно определить следующие основные подсистемы:

- **Модуль аутентификации и авторизации**  
  Обеспечивает регистрацию новых пользователей, вход в систему и выход из учётной записи. Для защиты ресурсов используется аутентификация на основе JSON Web Token (JWT): после успешного входа пользователь получает токен, который передаётся в заголовках последующих запросов. На сервере реализована проверка подлинности токена и разграничение доступа к ресурсам в соответствии с ролью пользователя.

- **Модуль управления пользователями и профилями**  
  Отвечает за хранение и изменение персональных данных пользователей. Для пользователей системы хранятся имя, фамилия, контактная информация, город, аватар и краткое описание профиля. На клиенте модуль представлен страницами личного кабинета и формами редактирования профиля, на сервере — набором API-эндпоинтов для чтения и обновления данных профиля.

- **Модуль мероприятий и заявок**  
  Реализует функциональность по созданию, редактированию, удалению и отображению волонтёрских мероприятий, а также управление заявками волонтёров на участие. Координатор формирует карточки мероприятий с указанием названия, описания, даты, времени, места проведения, категории и лимита участников. Волонтёры просматривают список доступных мероприятий, открывают страницу конкретного мероприятия, подают заявку на участие и при необходимости отменяют её. Состояние заявки отображается на странице мероприятия. Серверный слой обрабатывает соответствующие запросы, контролирует отсутствие повторной подачи заявки и соблюдение лимита участников, а также автоматически обновляет количество свободных мест. Для координатора также предусматривается просмотр списка пользователей, подавших заявки на конкретное мероприятие.

Опционально архитектура предусматривает возможность расширения за счёт модулей уведомлений, журналирования действий пользователей и подтверждения участия в мероприятиях, которые могут быть добавлены без изменения базовой трёхслойной структуры.

### 3. Взаимодействие компонентов

Взаимодействие между компонентами системы осуществляется следующим образом:

1. Пользователь работает с приложением через веб-браузер, в котором загружается клиентское React-приложение.  
2. При выполнении действий (регистрация, вход в систему, просмотр списка мероприятий, просмотр страницы мероприятия, подача заявки, отмена заявки, редактирование профиля, создание или редактирование мероприятия координатором) клиентское приложение отправляет HTTP-запросы к серверному API, например `POST /auth/register`, `POST /auth/login`, `GET /events`, `GET /events/{id}`, `POST /events/{id}/applications`, `DELETE /events/{id}/applications`, `PUT /profile`, `POST /coordinator/events`.  
3. Сервер приложений на базе Express принимает запрос, выполняет необходимые проверки (подлинность токена, права доступа, корректность входных данных), обращается к базе данных PostgreSQL для чтения или изменения данных и формирует ответ в формате JSON.  
4. Клиентское приложение получает ответ сервера и обновляет пользовательский интерфейс: изменяет состояние форм, отображает данные профиля, обновляет список мероприятий, показывает состояние заявки на странице мероприятия и, для координатора, отображает список пользователей, подавших заявки на участие.

Для иллюстрации такой схемы может быть использована диаграмма компонентов, на которой выделены следующие узлы:

- «Пользователь (браузер)» — источник запросов и получатель HTML/JS/CSS и данных;  
- «Клиентское приложение (React SPA)» — отвечает за интерфейс и отправку запросов к API;  
- «Сервер приложений (Express / Node.js)» — обрабатывает запросы, реализует бизнес-логику и обращается к данным;  
- «База данных (PostgreSQL)» — хранит доменные данные системы.

Между этими узлами отображаются направленные связи: от браузера к клиентскому приложению (загрузка интерфейса), от клиентского приложения к серверу (HTTP-запросы), от сервера к базе данных (SQL-запросы) и обратные ответы. Такая диаграмма наглядно демонстрирует распределение ролей между компонентами и поддерживает текстовое описание архитектуры.
