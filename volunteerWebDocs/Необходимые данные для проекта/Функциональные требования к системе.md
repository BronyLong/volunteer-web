# 1. Общие функциональные требования

## 1.1 Регистрация и аутентификация

Система должна обеспечивать:

1. Регистрацию пользователя по email и паролю.
2. Проверку уникальности email при регистрации.
3. Хранение пароля в виде криптографического хеша.
4. Авторизацию пользователя по email и паролю.
5. Выход из системы (инвалидация токена на клиенте).
6. Ограничение доступа к защищённым маршрутам только для авторизованных пользователей.
7. Разграничение доступа на основе ролей (RBAC) для ролей: гость, волонтёр, координатор, администратор.

## 1.2 Навигация и общие возможности интерфейса

Система должна обеспечивать:

1. Доступ к основным разделам через навигационное меню:
   - главная страница;
   - список мероприятий;
   - вход/регистрация;
   - личный кабинет (для авторизованных пользователей).
2. Корректное отображение страниц при отсутствии данных:
   - нет доступных мероприятий;
   - нет активных заявок;
   - нет начисленных часов.
3. Отображение страницы ошибки «404 / страница не найдена» при переходе на несуществующий маршрут.

# 2. Требования к функционалу по ролям

## 2.1 Гость (неавторизованный пользователь)

Система должна позволять гостю:

1. Просматривать главную страницу.
2. Просматривать список мероприятий.
3. Фильтровать мероприятия:
   - по дате;
   - по категории;
   - по статусу (открыто, завершено, отменено).
4. Просматривать подробную информацию о мероприятии.
5. Регистрироваться в системе.
6. Переходить к странице авторизации.

Система не должна позволять гостю:

- записываться на мероприятия;
- просматривать личный кабинет;
- создавать или редактировать мероприятия;
- просматривать административные разделы.

## 2.2 Волонтёр

Система должна обеспечивать волонтёру возможность.

### 2.2.1 Управление профилем

1. Просматривать личный профиль (личные данные, контакты, дата рождения).
2. Редактировать персональные данные (имя, фамилию, отчество, телефон, город и др.).
3. Загружать и изменять аватар.
4. Просматривать краткое описание (bio), если оно задано.
5. Просматривать историю участия в мероприятиях.

### 2.2.2 Работа с мероприятиями

6. Просматривать список доступных мероприятий.
7. Просматривать подробную информацию о выбранном мероприятии.
8. Подавать заявку на участие в мероприятии.
9. Отменять заявку до момента её подтверждения координатором.
10. Просматривать статус каждой своей заявки:
    - ожидание;
    - подтверждено;
    - отклонено.
11. Просматривать отдельный список «Мои мероприятия/мои заявки» в личном кабинете.

### 2.2.3 Учёт активности и статистика

12. Просматривать начисленные волонтёрские часы по каждому мероприятию.
13. Просматривать общее количество отработанных часов.
14. Просматривать количество мероприятий, в которых волонтёр принял участие.
15. Просматривать сводную статистику участия (динамика, последние мероприятия) в личном кабинете.

## 2.3 Координатор

Система должна обеспечивать координатору возможность.

### 2.3.1 Управление мероприятиями

1. Создавать новое мероприятие с указанием:
   - названия;
   - описания;
   - даты и времени проведения;
   - места проведения;
   - категории мероприятия;
   - лимита участников;
   - статуса мероприятия (открыто, завершено, отменено).
2. Редактировать данные существующего мероприятия.
3. Удалять мероприятие (при отсутствии критичных зависимостей или при наличии соответствующих правил).
4. Изменять статус мероприятия (например, переводить в состояние «завершено» или «отменено»).

### 2.3.2 Управление участниками

5. Просматривать список всех заявок волонтёров на конкретное мероприятие.
6. Подтверждать или отклонять заявки волонтёров.
7. Просматривать контактную информацию волонтёров, подавших заявку.
8. Подтверждать фактически отработанные часы по результатам проведения мероприятия.

### 2.3.3 Просмотр статистики по мероприятиям

10. Просматривать количество участников по каждому мероприятию.
11. Просматривать суммарное количество отработанных часов по мероприятию.
12. Просматривать список завершённых мероприятий и краткую статистику по ним.

## 2.4 Администратор

Система должна обеспечивать администратору возможность.

### 2.4.1 Управление пользователями

1. Просматривать список всех зарегистрированных пользователей.
2. Фильтровать и искать пользователей по email, ФИО, роли.
3. Изменять роли пользователей (волонтёр, координатор, администратор).
4. Блокировать аккаунты пользователей.
5. Разблокировать ранее заблокированные аккаунты.
6. Удалять пользователей (при необходимости и с учётом требований по хранению данных).

### 2.4.2 Управление мероприятиями

7. Просматривать список всех мероприятий в системе.
8. Назначать координаторов для мероприятий.
9. Удалять мероприятия (например, в случае ошибки создания).
10. Редактировать данные мероприятий при необходимости административного вмешательства.

### 2.4.3 Управление системой

11. Управлять категориями мероприятий (создание, редактирование, удаление).
12. Просматривать общую статистику системы:
    - количество пользователей по ролям;
    - количество активных и завершённых мероприятий;
    - суммарное количество отработанных часов.
13. Просматривать системные журналы (опционально), включая ключевые действия пользователей.

# 3. Требования к управлению мероприятиями

Система должна обеспечивать:

1. Хранение данных о мероприятии:
   - название;
   - короткое и полное описание;
   - дата и время начала (и при необходимости окончания);
   - место проведения;
   - лимит участников;
   - статус мероприятия (открыто, завершено, отменено);
   - категория мероприятия.
2. Отображение количества свободных мест для каждого мероприятия.
3. Автоматическое закрытие возможности регистрации при достижении установленного лимита участников.
4. Возможность изменения статуса мероприятия:
   - открыто;
   - завершено;
   - отменено.

# 4. Управление жизненным циклом мероприятия (опциональные расширения)

Дополнительно система может обеспечивать:

1. Перевод мероприятия в статус «архив» после его завершения.
2. Автоматическое закрытие мероприятия после даты проведения.
3. Возможность повторного проведения мероприятия (создание копии с новыми датами).
4. Клонирование мероприятия для ускорения создания похожих событий.

# 5. Требования к учёту волонтёрских часов

Система должна обеспечивать:

1. Начисление волонтёрских часов после подтверждения участия координатором.
2. Хранение информации о часах в разрезе мероприятий и волонтёров.
3. Автоматический расчёт общего количества часов для каждого волонтёра.
4. Отображение статистики часов в личном кабинете волонтёра.
5. Возможность корректировки количества часов координатором (при обнаружении ошибок), с фиксацией факта изменения в журнале (при наличии модуля журналирования).

# 6. Требования к поиску и фильтрации

Система должна обеспечивать:

1. Поиск мероприятий по названию.
2. Фильтрацию мероприятий:
   - по дате проведения;
   - по категории;
   - по статусу.
3. Сортировку мероприятий по дате проведения.
4. (Опционально) Фильтрацию мероприятий по доступности мест (есть свободные / набор закрыт).

# 7. Требования к журналированию действий

Система должна обеспечивать:

1. Фиксацию факта регистрации пользователей.
2. Фиксацию изменений ролей пользователей.
3. Фиксацию создания и удаления мероприятий.
4. Фиксацию подтверждения и отклонения заявок на участие.
5. Фиксацию начисления и изменения волонтёрских часов.
6. Хранение времени выполнения ключевых операций.

# 8. Модуль категорий и тегов

Система должна обеспечивать:

1. Создание категорий мероприятий.
2. Назначение категории мероприятию.
3. Фильтрацию мероприятий по категориям.
4. Управление категориями администратором (создание, редактирование, удаление).

# 9. Настройки профиля

Система должна обеспечивать пользователю возможность:

1. Изменять пароль.
2. Удалять свой аккаунт (с учётом требований по хранению данных).
3. Настраивать видимость отдельных полей профиля (опционально, при необходимости).

# 10. Модуль новостей (опционально)

При наличии ресурсов для реализации система может обеспечивать следующий функционал новостного раздела:

1. Создание новостных материалов (координаторами или администраторами).
2. Редактирование и удаление новостных материалов уполномоченными пользователями.
3. Просмотр списка новостей всеми пользователями (включая гостей).
4. Просмотр детальной страницы новости.
5. Фильтрацию и сортировку новостей по дате (опционально).

# 11. Геймификация (опционально)

Система может обеспечивать:

1. Присвоение уровней волонтёрам на основе количества отработанных часов или числа мероприятий.
2. Выдачу достижений за выполнение определённых условий (например, участие в N мероприятиях).
3. Формирование рейтинга активных участников.
4. Отображение бейджей в профиле волонтёра.

# 12. Административные функции (обобщение)

Система должна обеспечивать административный уровень доступа, включающий:

1. Просмотр заблокированных пользователей.
2. Разблокировку аккаунтов.
3. Просмотр истории действий пользователя (при наличии модуля журналирования).
4. Массовое назначение ролей (опционально, для упрощения администрирования).
