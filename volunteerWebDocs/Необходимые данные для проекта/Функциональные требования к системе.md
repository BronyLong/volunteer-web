# 1. Общие функциональные требования

## 1.1 Регистрация и аутентификация

Система должна обеспечивать:

1. Регистрацию пользователя по email и паролю.
2. Проверку уникальности email при регистрации.
3. Авторизацию пользователя по email и паролю.
4. Ограничение доступа к защищённым маршрутам только для авторизованных пользователей.
5. Разграничение доступа на основе ролей (RBAC) для ролей: гость, волонтёр, координатор.

## 1.2 Навигация и общие возможности интерфейса

Система должна обеспечивать:

1. Доступ к основным разделам через навигационное меню:
   - главная страница;
   - список мероприятий;
   - вход/регистрация;
   - личный кабинет (для авторизованных пользователей).
   - Страницы управления мероприятиями (для координатора)
2. Корректное отображение страниц при отсутствии данных:
   - нет доступных мероприятий;
3. Отображение страницы ошибки «404 / страница не найдена» при переходе на несуществующий маршрут.
4. Нет поданных заявок на мероприятие (для страницы мероприятия координатора) 

# 2. Требования к функционалу по ролям

## 2.1 Гость (неавторизованный пользователь)

Система должна позволять гостю:

1. Просматривать главную страницу.
2. Просматривать список мероприятий.
3. Фильтровать мероприятия по категории.
4. Просматривать подробную информацию о мероприятии.
5. Регистрироваться в системе.
6. Переходить к странице авторизации.

Система не должна позволять гостю:

- записываться на мероприятия;
- просматривать личный кабинет;
- создавать или редактировать мероприятия.

## 2.2 Волонтёр

Система должна обеспечивать волонтёру возможность.

### 2.2.1 Управление профилем

1. Просматривать личный профиль (личные данные, контакты, город, ссылки на социальные сети).
2. Редактировать персональные данные (имя, фамилию, телефон, город и др.).
3. Загружать и изменять аватар.
4. Просматривать краткое описание (bio), если оно задано.

### 2.2.2 Работа с мероприятиями

5. Просматривать список доступных мероприятий.
6. Просматривать подробную информацию о выбранном мероприятии.
7. Подавать заявку на участие в мероприятии.
8. Отменять заявку на участие в мероприятии.
9. Отображение состояния заявки на странице мероприятия:
    - «заявка не подана»
    - «заявка отправлена»
10. Запрет повторной подачи заявки

## 2.3 Координатор

Система должна обеспечивать координатору возможность.

### 2.3.1 Управление мероприятиями

1. Создавать новое мероприятие с указанием:
   - названия;
   - описания;
   - даты и времени проведения;
   - места проведения;
   - категории мероприятия;
   - лимита участников;
2. Редактировать данные существующего мероприятия.
3. Удалять существующее мероприятие.
4. Просмотр списка пользователей, подавших заявки на участие в мероприятии

# 3. Требования к управлению мероприятиями

Система должна обеспечивать:

1. Хранение данных о мероприятии:
   - название;
   - описание;
   - дата и время начала;
   - место проведения;
   - лимит участников;
   - категория мероприятия.
2. Отображение количества свободных мест для каждого мероприятия.
3. Автоматическое закрытие возможности регистрации при достижении установленного лимита участников.
4. Автоматическое уменьшение количества доступных мест при подаче заявки;
5. Автоматическое увеличение количества доступных мест при отмене заявки;

# 4. Требования к поиску и фильтрации

Система должна обеспечивать:

1. Фильтрацию мероприятий по категории.
2. Сортировку мероприятий по дате проведения (сначала отображаются ближайшие мероприятия).

# 5. Требования к журналированию действий

Система должна обеспечивать:

1. Фиксацию факта регистрации пользователей.
2. Фиксацию создания и удаления мероприятий.
3. Фиксацию редактирования информации о мероприятии.
4. Фиксацию подачи заявок на участие.
5. Хранение времени выполнения ключевых операций.

## 6. Требования к валидации данных

Система должна обеспечивать:

- обязательность полей при регистрации;
- проверка формата email;
- обязательность полей при создании мероприятия;
