# 1. Общие функциональные требования
## 1.1 Регистрация и аутентификация

Система должна обеспечивать:
1. Регистрацию пользователя по email и паролю.
2. Проверку уникальности email.
3. Хранение пароля в виде хеша.
4. Авторизацию пользователя.
5. Выход из системы.
6. Восстановление пароля.
7. Подтверждение email (опционально).
8. Ограничение доступа к защищённым маршрутам.
9. Разграничение доступа на основе ролей (RBAC). 
# 2. Требования к функционалу по ролям
# 2.1 Гость (неавторизованный пользователь)

Система должна позволять гостю:
1. Просматривать главную страницу.
2. Просматривать список мероприятий.
3. Фильтровать мероприятия:
    - по дате
    - по категории
    - по статусу.
4. Просматривать подробную информацию о мероприятии.
5. Зарегистрироваться в системе.
6. Перейти к странице авторизации.

Система не должна позволять гостю:
- записываться на мероприятия;
- просматривать личный кабинет;
- создавать мероприятия.
# 2.2 Волонтёр

Система должна обеспечивать волонтёру возможность:
## 2.2.1 Управление профилем

1. Просматривать личный профиль.
2. Редактировать персональные данные.
3. Загружать и изменять аватар.
4. Просматривать историю участия.
## 2.2.2 Работа с мероприятиями

5. Просматривать список доступных мероприятий.
6. Записываться на мероприятие.
7. Отменять запись (до подтверждения).
8. Просматривать статус заявки:
    - ожидание
    - подтверждено
    - отклонено.
## 2.2.3 Учёт активности

9. Просматривать начисленные часы.
10. Просматривать общее количество часов.
11. Просматривать статистику участия.

# 2.3 Координатор

Система должна обеспечивать координатору возможность:
## 2.3.1 Управление мероприятиями

1. Создавать мероприятие.
2. Редактировать мероприятие.
3. Удалять мероприятие.
4. Устанавливать лимит участников.
5. Назначать дату, время и место проведения.
6. Указывать категорию мероприятия.
## 2.3.2 Управление участниками

7. Просматривать список зарегистрированных волонтёров.
8. Подтверждать или отклонять заявки.
9. Назначать задачи участникам.
10. Подтверждать фактически отработанные часы.
## 2.3.3 Просмотр статистики

11. Просматривать количество участников.
12. Просматривать суммарные часы по мероприятию.

# 2.4 Администратор

Система должна обеспечивать администратору возможность:
## 2.4.1 Управление пользователями

1. Просматривать список всех пользователей.
2. Изменять роли пользователей.
3. Блокировать аккаунты.
4. Удалять пользователей.
## 2.4.2 Управление мероприятиями

5. Просматривать все мероприятия.
6. Назначать координаторов.
7. Удалять мероприятия.
8. Редактировать данные мероприятий.

## 2.4.3 Управление системой

9. Управлять категориями мероприятий.
10. Просматривать общую статистику системы.
11. Просматривать системные журналы (опционально).

# 3. Требования к управлению мероприятиями

Система должна обеспечивать:

1. Хранение данных о мероприятии:
    - название
    - описание
    - дата и время
    - место проведения
    - лимит участников
    - статус мероприятия.
        
2. Отображение количества свободных мест.    
3. Автоматическое закрытие регистрации при достижении лимита.
4. Возможность изменения статуса мероприятия:    
    - открыто
    - завершено
    - отменено.
# 4. Управление жизненным циклом мероприятия

Дополнительно можно добавить:

1. Перевод мероприятия в статус "архив".
2. Автоматическое закрытие мероприятия после даты проведения.
3. Возможность повторного проведения мероприятия.
4. Клонирование мероприятия.
# 5. Требования к учёту волонтёрских часов

Система должна обеспечивать:

1. Начисление часов после подтверждения координатором.
2. Хранение часов в разрезе мероприятий.    
3. Автоматический расчёт общего количества часов.
4. Отображение статистики в личном кабинете.    
# 6. Требования к поиску и фильтрации

Система должна обеспечивать:

1. Поиск мероприятий по названию.
2. Фильтрацию по дате.
3. Фильтрацию по категории.
4. Сортировку по дате проведения.
# 7. Требования к журналированию действий

Система должна обеспечивать:

1. Фиксацию регистрации пользователей.
2. Фиксацию изменений ролей.
3. Фиксацию создания и удаления мероприятий.
4. Хранение времени выполнения ключевых операций.
# 8. Геймификация (опционально)

Система может обеспечивать:

1. Присвоение уровней волонтёрам.
2. Выдачу достижений.
3. Формирование рейтинга активных участников.
4. Отображение бейджей в профиле.
# 9. Модуль категорий и тегов

Система должна обеспечивать:

1. Создание категорий мероприятий.
2. Назначение категории мероприятию.
3. Фильтрацию по категориям.
4. Управление категориями администратором.
# 10. Настройки профиля

1. Изменение пароля.
2. Удаление аккаунта.
3. Настройка видимости профиля.
# 11. Административные функции

1. Просмотр заблокированных пользователей.
2. Разблокировка аккаунтов.
3. Просмотр истории действий пользователя.
4. Массовое назначение ролей.
