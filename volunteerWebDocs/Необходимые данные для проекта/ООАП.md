# Объектно-ориентированный анализ и проектирование
## 1. Общая характеристика объектно-ориентированного подхода

При разработке программных систем одним из наиболее распространённых подходов является объектно-ориентированное проектирование. Данный подход основывается на представлении программной системы в виде совокупности взаимосвязанных объектов, каждый из которых обладает собственными свойствами и поведением.

Объектно-ориентированный анализ и проектирование представляет собой методологию разработки программного обеспечения, направленную на выявление сущностей предметной области, определение их характеристик и формирование структуры программной системы на основе классов и объектов. Такой подход позволяет более естественно моделировать реальные процессы и упрощает дальнейшее развитие и сопровождение программного продукта.

Основными преимуществами объектно-ориентированного подхода являются:

- высокая степень модульности системы;
- возможность повторного использования компонентов;
- упрощение сопровождения программного кода;
- повышение масштабируемости системы;
- более наглядное представление структуры программного решения.

В рамках разработки веб-приложения для автоматизации деятельности волонтёрской организации объектно-ориентированный подход применяется для моделирования основных сущностей системы, таких как пользователи, мероприятия, заявки на участие и учёт волонтёрских часов.
# 2. Объектно-ориентированный анализ системы

Объектно-ориентированный анализ направлен на исследование предметной области и выявление ключевых сущностей, участвующих в функционировании системы. На данном этапе определяется перечень объектов, их основные характеристики и взаимосвязи между ними.

Анализ бизнес-процессов волонтёрской организации позволил выделить следующие основные сущности системы:

- пользователь системы;  
- профиль пользователя;
- волонтёрское мероприятие;
- заявка на участие в мероприятии;
- учёт волонтёрских часов;
- категория мероприятия.

Каждая из указанных сущностей представляет собой отдельный объект предметной области и обладает собственными атрибутами и поведением.
## 2.1 Объект «Пользователь»

Объект «Пользователь» представляет собой зарегистрированного участника системы. Пользователи могут иметь различные роли, определяющие их права доступа и возможности взаимодействия с системой.

Основными ролями пользователей являются:

- волонтёр;    
- координатор;
- администратор.

Основные атрибуты объекта «Пользователь»:

- уникальный идентификатор пользователя;    
- адрес электронной почты;    
- хеш пароля;    
- роль пользователя;
- дата создания учетной записи;    
- статус активности учетной записи;    
- время последнего входа в систему.

Данный объект обеспечивает выполнение следующих функций:

- регистрация пользователя в системе;    
- авторизация пользователя;    
- управление доступом к функциональным возможностям системы;    
- хранение основной информации об учетной записи.    

## 2.2 Объект «Профиль пользователя»

Профиль пользователя содержит расширенную информацию о зарегистрированном участнике системы. Эти данные используются координаторами и администраторами для взаимодействия с волонтёрами и формирования списков участников мероприятий.

Основные атрибуты профиля пользователя:

- имя;
- фамилия;
- отчество;
- номер телефона;
- город проживания;
- дата рождения;
- фотография профиля;
- краткая информация о пользователе.

Профиль пользователя может редактироваться самим пользователем через личный кабинет системы.
## 2.3 Объект «Мероприятие»

Объект «Мероприятие» описывает волонтёрскую активность, проводимую организацией. Мероприятия создаются координаторами и доступны для просмотра и регистрации волонтёров.

Основные атрибуты объекта «Мероприятие»:

- идентификатор мероприятия;
- название мероприятия;
- описание мероприятия;
- дата и время проведения;
- место проведения;
- категория мероприятия;
- лимит участников;
- статус мероприятия.

Основные функции объекта:

- создание мероприятия;
- изменение параметров мероприятия;
- управление статусом мероприятия;
- хранение информации о проводимых событиях.
## 2.4 Объект «Заявка на участие»

Объект «Заявка на участие» отражает факт подачи волонтёром заявки на участие в конкретном мероприятии.

Основные атрибуты объекта:

- идентификатор заявки;
- идентификатор пользователя;
- идентификатор мероприятия;
- статус заявки;
- дата создания заявки;
- дата изменения статуса заявки.

Статус заявки может принимать следующие значения:

- ожидание рассмотрения;
- подтверждено;
- отклонено.

Заявки рассматриваются координаторами мероприятий, которые принимают решение о подтверждении или отклонении участия волонтёра.
## 2.5 Объект «Учёт волонтёрских часов»

Данный объект предназначен для фиксации фактически отработанного времени волонтёров на мероприятиях.

Основные атрибуты объекта:

- идентификатор записи;
- идентификатор пользователя;
- идентификатор мероприятия;
- количество отработанных часов;
- идентификатор координатора, подтвердившего участие;
- дата подтверждения.

На основе этих данных система формирует статистику активности волонтёров и позволяет отслеживать их вклад в деятельность организации.
## 2.6 Объект «Категория мероприятия»

Категории используются для классификации мероприятий по тематике.

Примеры категорий:

- помощь пожилым людям;
- экологические акции;
- помощь животным;
- социальные проекты.

Использование категорий позволяет организовать удобный поиск и фильтрацию мероприятий в системе.
# 3. Объектно-ориентированное проектирование системы

На этапе объектно-ориентированного проектирования определяется структура классов программной системы, их свойства, методы и связи между ними.

Основными классами разрабатываемой системы являются:

- User
- Profile
- Event
- Application
- VolunteerHours
- Category

Каждый из классов содержит набор атрибутов и методов, реализующих бизнес-логику системы.
## 3.1 Класс User

Класс User описывает учетную запись пользователя.

Основные атрибуты:

- id
- email
- password_hash
- role
- created_at
- is_active

Основные методы:

- register()
- login()
- logout()
- changePassword()
## 3.2 Класс Event

Класс Event описывает волонтёрское мероприятие.

Основные атрибуты:

- id
- title
- description
- date
- location
- participant_limit
- status

Основные методы:

- createEvent()
- updateEvent()
- changeStatus()
- deleteEvent()
## 3.3 Класс Application

Класс Application представляет заявку пользователя на участие в мероприятии.

Основные атрибуты:

- id
- user_id
- event_id
- status

Основные методы:

- applyForEvent()
- approveApplication()
- rejectApplication()
- cancelApplication()
## 3.4 Класс VolunteerHours

Класс VolunteerHours отвечает за учет отработанных часов.

Основные атрибуты:

- id
- user_id
- event_id
- hours_worked
- confirmed_by

Основные методы:

- addHours()
- updateHours()
- getVolunteerStatistics()
# 4. Связи между объектами системы

В процессе проектирования определяются взаимосвязи между основными объектами системы.

Основные связи:

- один пользователь может подать несколько заявок на участие;
- одно мероприятие может иметь множество заявок;
- один пользователь может участвовать в нескольких мероприятиях;
- одно мероприятие может иметь несколько записей учёта часов;
- одна категория может быть связана с несколькими мероприятиями.

Таким образом формируется структура взаимодействия объектов системы.
